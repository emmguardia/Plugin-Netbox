# Test harness

Spins up a real NetBox stack (NetBox + PostgreSQL + Redis) in Docker and runs
the plugin against it. Used to validate the plugin before deploying it to a
real environment. Everything here is for testing only — none of it ships with
the plugin.

## What gets tested

- **Plugin loads** in a real NetBox (`manage.py check`).
- **Migration** is generated by NetBox itself and applies cleanly to PostgreSQL.
- **16 Django tests** (`netbox_k3s/tests/test_plugin.py`): models, URL reversing,
  UI list/detail/add views (incl. custom templates with nested tables), REST API
  CRUD, filtering, search, and unauthenticated access denial.
- **8 sync unit tests** (`scripts/test_sync.py`): kubectl→payload transformation,
  PATCH-vs-POST upsert logic.
- **End-to-end sync**: the real `sync_k3s.py` runs against a live NetBox web
  server using a fake `kubectl`, then idempotency is verified (a second run
  creates no duplicates).

## Run it

```powershell
# 1. Start the stack
docker compose -f test/docker-compose.yml up -d

# 2. Install the plugin (PYTHONPATH) and apply migrations
docker compose -f test/docker-compose.yml exec -e PYTHONPATH=/source -w /opt/netbox/netbox `
  netbox /opt/netbox/venv/bin/python manage.py migrate

# 3. Django test suite
docker compose -f test/docker-compose.yml exec -e PYTHONPATH=/source -w /opt/netbox/netbox `
  netbox /opt/netbox/venv/bin/python manage.py test netbox_k3s

# 4. Sync unit tests
docker compose -f test/docker-compose.yml cp ../scripts/. netbox:/tmp/   # or docker cp
docker compose -f test/docker-compose.yml exec -w /tmp netbox /opt/netbox/venv/bin/python test_sync.py

# 5. Tear down
docker compose -f test/docker-compose.yml down -v
```

> The plugin source is mounted into the container at `/source` and made
> importable via `PYTHONPATH`, so code changes are picked up without rebuilding
> the image. `DEVELOPER = True` and `API_TOKEN_PEPPERS` in `plugins.py` exist
> only to enable `makemigrations` and v2 API-token creation inside the tests.

## Notes discovered while testing

- `netboxcommunity/netbox:latest` reports image version `5.0.1` in
  `/opt/netbox/VERSION`, but the **NetBox application version is 4.6.2**
  (`manage.py version`). The plugin's `min/max_version` target the app version.
- NetBox 4.6 has **no `pip`** in its venv (it uses `uv`). For production, install
  via a derived image or `local_requirements.txt` (see `docs/INSTALL.md`).
- NetBox 4.6 uses **v2 API tokens**. The classic `Authorization: Token <value>`
  scheme still works — you just pass whatever token string NetBox shows you.
